## 2. BUG-01：凭据明文落库、返回给 viewer、配置文件权限过宽

### 2.1 根因

`generateConfig()` 为 Claude、Gemini、OpenCode 生成的原生配置包含 API Key。随后：

1. `createAgentTask()` 把完整 `content` 写入 `agent_tasks.payload_json`；
2. `audit()` 把完整 payload 展开写入 `audit_log.after_json`；
3. Agent 的 `handleWriteConfig()` 把完整内容放进任务结果；
4. `handleTaskResult()` 又把结果写入 `result_json` 和 `configs.content`；
5. `/api/servers/:id/tasks` 和 `/api/audit` 对 viewer 开放；
6. Agent 用 `0644` 写配置和备份，含 Key 的文件对同机其他用户可读。

迁移 008 只清理旧 `set_credential.credentials`，没有覆盖 `write_config.content`、`result_json` 或 `configs.content`。

### 2.2 目标设计

将“任务持久化描述”和“实际下发内容”分开：

```text
DB 中的任务（仅引用）
  tool + provider_id/key_id/model_id
                 │
                 ▼ dispatch 时
materializeTaskPayload() 解密 provider key、生成配置
                 │
                 ▼ TLS/WS
Agent 内存中的明文配置 → 0600 文件
```

数据库中允许保存：provider/key/model 引用、格式、目标路径、备份名、内容哈希、执行状态。数据库中禁止保存：API Key、完整敏感配置、shell credential 内容。

### 2.3 后端具体改法

#### 2.3.1 引入安全的任务描述

在 `console/apps/api/src/modules/agent/routes.ts` 定义两类 payload：

```ts
type ProviderConfigRef = {
  tool: "codex" | "claude" | "gemini" | "opencode"
  source: "provider_refs"
  entries: Array<{
    provider_id: string
    key_id: string
    model_id: string
    primary?: boolean
  }>
}

type ManualConfigRef = {
  tool: "codex" | "claude" | "gemini" | "opencode"
  source: "encrypted_manual"
  format: string
}
```

批量下发和 ServerDetail 的供应商下发不再提前调用 `generateConfig()` 并创建 `{content}` 任务，而是只创建 `ProviderConfigRef`。在 `materializeTaskPayload()` 中：

1. 逐个校验 provider/key/model 的归属和 enabled 状态；
2. 解密 provider key；
3. 调用 `generateConfig()` 或 `mergeOpenCodeConfig()`；
4. 只把生成出的 `{tool, format, content}` 发送给 Agent；
5. 不把 materialized payload 回写数据库或日志。

建议保留 action 名 `write_config`，通过 `payload.source` 区分引用型任务，避免同时修改 Agent action 路由。

#### 2.3.2 手工配置内容加密存储

手工编辑配置无法转换成 provider 引用，需要加密。新增迁移 `012_encrypt_sensitive_task_content.sql`：

```sql
ALTER TABLE agent_tasks ADD COLUMN encrypted_payload TEXT;
ALTER TABLE agent_tasks ADD COLUMN encrypted_payload_iv TEXT;

ALTER TABLE configs ADD COLUMN encrypted_content TEXT;
ALTER TABLE configs ADD COLUMN encrypted_content_iv TEXT;
ALTER TABLE configs ADD COLUMN content_sha256 TEXT;
```

同步更新 `schema.sql`。`payload_json` 和 `configs.content` 暂时保留，以兼容 SQLite ALTER 限制和旧代码，但敏感记录只写占位值：

```json
{"tool":"opencode","source":"encrypted_manual","redacted":true}
```

扩展 `createAgentTask()`：

```ts
function createAgentTask(
  serverId: string,
  userId: string,
  action: string,
  payload: unknown,
  opts?: {
    sensitivePayload?: unknown
    auditMeta?: Record<string, unknown>
  }
)
```

当存在 `sensitivePayload` 时，使用现有 `encrypt(JSON.stringify(...))` 写入 `encrypted_payload/encrypted_payload_iv`。`pushTaskToAgent()` 和 REST claim 在 materialize 前解密；解密失败必须把任务置为 failed，不能发送空内容。

#### 2.3.3 审计改为白名单元数据

禁止以下写法：

```ts
audit(userId, "agent_task.create", target, null, {
  server_id: serverId,
  action,
  ...(payload as any),
})
```

改成明确白名单：

```ts
audit(userId, "agent_task.create", `agent_task:${id}`, null, {
  server_id: serverId,
  action,
  tool: safeMeta.tool,
  provider_ids: safeMeta.provider_ids,
  key_ids: safeMeta.key_ids,
  model_ids: safeMeta.model_ids,
  sensitive: Boolean(opts?.sensitivePayload),
})
```

审计层再增加防御性递归脱敏，键名匹配以下模式时写 `"[REDACTED]"`：

```text
api_key, apiKey, token, secret, password, authorization,
ANTHROPIC_AUTH_TOKEN, GEMINI_API_KEY, OPENAI_API_KEY
```

这层只做兜底，不能代替调用点白名单。

#### 2.3.4 任务结果不再回显 write_config.content

修改 Agent `handleWriteConfig()`，成功结果只返回：

```go
return map[string]interface{}{
    "path": path,
    "format": p.Format,
    "content_sha256": fingerprint(p.Content),
    "backup": backupName,
}, ""
```

`handleTaskResult()` 对 `write_config` 即使收到旧 Agent 返回的 `result.content` 也必须删除后再持久化：

```ts
const persistedResult = { ...result }
if (task.action === "write_config") delete persistedResult.content
```

`read_config` 返回的内容必须在 API 端立即加密后写 `configs.encrypted_content`，`result_json` 只保存元数据和 `config_version`。

#### 2.3.5 增加专用配置读取接口

新增：

```text
GET /api/servers/:id/configs/latest?tool=opencode
```

要求 operator 或 admin。接口读取 `configs.encrypted_content`、解密并返回给配置编辑器。viewer 只能看版本号、更新时间、hash，不能看到内容。

`GET /api/servers/:id/tasks` 对所有角色都只返回结构化安全字段，移除原始 `payload_json`/`result_json`，或在服务端解析后按 allowlist 组装：

```ts
{
  id, action, tool, status, error,
  created_at, claimed_at, finished_at,
  result: { path, format, backup, content_sha256 }
}
```

### 2.4 历史数据清理

SQL 无法用应用的 AES Key 加密历史内容。建议在迁移 012 后执行一次 TypeScript 启动清理函数：

1. 备份数据库；
2. 对能识别的 `configs.content` 使用 `encrypt()` 原地迁入加密列；
3. 把旧 `content` 改成 `[ENCRYPTED]`；
4. 对 `agent_tasks` 的 write/read config payload/result 做同样迁移或直接脱敏；
5. 对审计日志只做脱敏，不保留敏感原文；
6. 写入独立的一次性维护版本表，保证幂等。

不能仅通过 `LIKE '%apiKey%'` 判断，因为 raw config 可能使用自定义字段。对 `write_config` 和敏感工具的 config 快照应按 action/tool 全量处理。

### 2.5 Agent 文件权限

修改 `agent/internal/agent/agent.go`：

- `~/.ai-console-agent`、`creds` 目录使用 `0700`；
- 所有工具配置和配置备份使用 `0600`；
- 写入后显式 `os.Chmod(path, 0600)`，防止覆盖已存在的宽权限文件时权限不变；
- 用同目录临时文件 `0600` + `fsync` + `rename` 原子替换，避免截断后进程崩溃留下半份配置。

### 2.6 验收

- 创建 Claude/Gemini/OpenCode 单机与批量下发任务后，数据库全文搜索测试 Key，结果必须为 0。
- viewer 请求 task/audit/config 内容接口，分别确认不含明文或返回 403。
- Agent 端 `stat` 配置与备份，权限必须为 `600`，目录为 `700`。
- 旧数据库升级前后随机抽查配置可正常解密；错误 MASTER_KEY 必须明确报错，不能返回空配置。

## 3. BUG-02：viewer 可以调用导入回滚删除数据

### 3.1 立即修复

在 `console/apps/api/src/middleware/auth.ts` 的 `adminOnly` 中加入：

```ts
const adminOnly = [
  "/api/providers/import/",
  "/api/import-jobs/",
  "/api/agent/enroll-tokens",
]
```

这会让 `/api/import-jobs/:id/rollback` 的非 GET 请求仅 admin 可用。导入本身是 admin-only，回滚权限不应低于导入权限。

### 3.2 结构性修复

URL 前缀 RBAC 容易再次漏掉新路由。后续改为路由级权限元数据：

```ts
app.post("/api/import-jobs/:id/rollback", {
  config: { minRole: "admin" },
}, handler)
```

全局 hook 从 `req.routeOptions.config.minRole` 读取要求。没有明确 `minRole` 的 `/api/*` 非公开路由在启动时记录 warning；所有 mutation 路由必须显式声明角色。

为避免一次重构改动过大，先提交前缀补丁，再单独提交 route metadata 重构。

## 4. BUG-03：并发刷新会删除刚轮换成功的 refresh Cookie

### 4.1 根因

`rotateSession()` 用 `null` 同时表达三种情况：无效、replay、最近刚被另一个请求轮换。`/refresh` 无法区分，统一调用 `clearRefreshCookie()`。并发落败请求的过期 Set-Cookie 可能覆盖成功请求的新 Cookie。

### 4.2 服务端改法

将返回值改成判别联合：

```ts
type RotateResult =
  | { kind: "ok"; session: { accessToken: string; user: AuthUser } }
  | { kind: "recent_rotation" }
  | { kind: "invalid" }
  | { kind: "replay" }
```

处理规则：

- hash 匹配：正常轮换，返回 `ok`；
- hash 不匹配且 `rotated_at < 30s`：返回 `recent_rotation`，绝不清 Cookie；
- session 不存在或已 revoked：返回 `invalid`，清 Cookie；
- 超出宽限期的 hash 不匹配：撤销 session，返回 `replay` 并清 Cookie。

`POST /api/auth/refresh`：

```ts
if (result.kind === "ok") return result.session
if (result.kind === "recent_rotation") {
  return reply
    .header("retry-after", "0")
    .code(409)
    .send({ error: "refresh_already_rotated" })
}
clearRefreshCookie(reply)
return reply.code(401).send({ error: "unauthorized" })
```

不要在 `recent_rotation` 分支签发 access token，否则持有旧 refresh token 的攻击者可在宽限期内换取新 access token。

### 4.3 简化 `/auth/me`

`GET /api/auth/me` 只接受 access JWT，不在该 GET 路由中隐式轮换 refresh token：

```ts
const user = authFromRequest(req)
if (!user) return reply.code(401).send({ error: "unauthorized" })
return { user }
```

所有续期只经 `/auth/refresh`，让前端 single-flight 能覆盖 `me`、WS ticket 和普通 API 的 401。

### 4.4 前端跨标签页重试

单个标签页的 `refreshPromise` 无法协调多个标签页。`rawRequest()` 应暴露 HTTP status 和结构化 error code，不能只拼成字符串。

当 `/refresh` 返回 `409 refresh_already_rotated` 时：

1. 不清 access token；
2. 等待 50ms、150ms、300ms 后最多重试 3 次；
3. 重试会使用浏览器 Cookie jar 中胜方已设置的新 Cookie；
4. 只有最终 401 才执行 `setAccessToken(null)`。

可选增强：使用 `BroadcastChannel("ai-console-auth")` 广播新 access token，减少多标签页重复 refresh。不要通过 BroadcastChannel 传播 refresh token。

### 4.5 验收

- 两个标签页用同一 refresh Cookie 同时请求 refresh，最终 Cookie 仍存在且至少一个请求成功；另一个返回 409 后重试成功。
- 页面在 access token 过期时同时发起 `me` 和 WS ticket，网络面板中只出现一次有效轮换链路。
- 用超过 30 秒的旧 refresh token重放，session 被撤销并产生 `auth.replay_detected` 审计。

## 5. BUG-04：批量回滚状态与真实远程状态不一致

### 5.1 状态机约束

只允许终态批次开始回滚：

```text
running ──▶ done ───────▶ rolling_back ──▶ rolled_back
    │         │                    └─────▶ partial_rollback
    └──────▶ partial ───▶ rolling_back
```

禁止：

- `running` 时回滚；
- `rolling_back` 时重复创建 restore task；
- `rolled_back`/`partial_rollback` 再次回滚；
- 缺少精确 backup 名时回退到“mtime 最新备份”。

### 5.2 数据结构

新增迁移 `013_batch_rollback_progress.sql`：

```sql
ALTER TABLE batch_jobs ADD COLUMN rollback_progress_json TEXT;
ALTER TABLE batch_jobs ADD COLUMN rollback_started_at INTEGER;
```

保留 `progress_json` 作为原始写任务记录，不再覆盖。`rollback_progress_json` 保存：

```json
[
  {
    "server_id": "...",
    "write_task_id": "...",
    "restore_task_id": "...",
    "backup": ".opencode.json.bak....",
    "state": "pending",
    "error": null
  }
]
```

### 5.3 API 实现

`POST /api/batch/:id/rollback` 首先做 CAS：

```sql
UPDATE batch_jobs
SET status='rolling_back', rollback_started_at=?
WHERE id=? AND status IN ('done','partial');
```

`changes===0` 时根据当前状态返回：

- `running`：409 `batch_not_finished`；
- `rolling_back`：202，返回现有进度，不创建新任务；
- `rolled_back/partial_rollback`：409 `already_rolled_back`。

创建 restore task 前逐项校验：

1. 原 write task 必须为 done；
2. `result_json.backup` 必须存在且是合法 bare filename；
3. 没有成功写入的 entry 标成 skipped，不创建 restore；
4. 没有任何可恢复 entry 时，批次进入 `partial_rollback` 或 `rollback_failed`，不能用 `every([]) === true` 判成功。

### 5.4 避免事务内提前推送任务

当前 `createAgentTask()` 会立即 push，调用方却可能仍处于 SQLite transaction。拆成：

```ts
insertAgentTask(...)       // 只插 DB，不发送
dispatchAgentTask(taskId)  // COMMIT 后发送
```

回滚流程：事务内插入所有 restore tasks 和 `rollback_progress_json`，提交后逐个 dispatch。批量执行原始 write tasks 也应使用同一模式，避免事务回滚后 Agent 已执行远程写入。

### 5.5 进度收敛

`broadcastBatchProgressForTask()`：

- running/done/partial 读取 `progress_json`；
- rolling_back 读取 `rollback_progress_json`；
- 每次更新都把新进度持久化，而不只是广播；
- restore 全 done 才是 rolled_back；任一 failed 则 partial_rollback；
- 空 rollback progress 永远不能判 rolled_back。

### 5.6 前端

- 回滚按钮只在 `done` 或 `partial` 显示；
- `rolling_back` 显示独立进度，禁止再次点击；
- `partial_rollback` 展示失败机器和错误；
- 409 使用明确 toast，不使用通用 alert。

### 5.7 验收

- 批次仍 running 时直接调用 rollback，必须 409，原任务不受影响。
- 连续两次调用 rollback，只能创建一组 restore task。
- 删除一个精确 backup 后回滚，该机器必须 failed，不能恢复其他“最新备份”。
- 0 个成功 write 的 partial batch 回滚不能显示 rolled_back。
- 模拟事务插入中途异常，Agent 不应收到任何未提交 task。

## 6. BUG-05：任务租约重领、旧结果覆盖、Agent 并发写配置

### 6.1 协议必须回传 nonce

WS `cmd_result` 增加顶层 nonce：

```json
{
  "type": "cmd_result",
  "id": "task-id",
  "nonce": "claim-nonce",
  "status": "ok",
  "payload": { "result": {} }
}
```

REST claim 返回 `{id, action, nonce, expires_at, payload}`；REST result body 同样增加 nonce。

服务端结果 CAS：

```sql
UPDATE agent_tasks
SET status=?, result_json=?, error=?, finished_at=?
WHERE id=? AND server_id=? AND status='running' AND nonce=?;
```

旧 nonce 返回 409 `stale_task_lease`，不能把新一轮 claim 标成 done。

灰度兼容：部署顺序必须先升级 Console，使其暂时接受旧 Agent 的无 nonce 结果；待所有 Agent 到新版本后，通过最小 Agent 版本门槛开启强校验并删除兼容分支。兼容窗口内不要启用激进 requeue。

### 6.2 租约续期

固定 5 分钟不适合升级等长任务。增加：

```text
POST /agent/tasks/:id/lease
```

body 为 `{nonce}`，服务端只在 `id/server/status/nonce` 全匹配时把 `expires_at` 延长 2 分钟。WS 可增加 `cmd_lease` 消息走同一函数。

Agent 执行任务期间每 45 秒续租；任务完成或 context 取消后停止 ticker。reaper 只回收没有续租的任务。

### 6.3 重试上限

`attempt_count` 在 claim 成功时 `+1`。reaper：

- `attempt_count < 3`：清 claim 后回 pending；
- `attempt_count >= 3`：置 failed，错误为 `task lease expired after 3 attempts`；
- 进入 failed 后推动 batch 状态收敛。

### 6.4 Agent 本地去重日志

仅校验 nonce 能防旧结果覆盖，不能防远程副作用重复发生。Agent 增加 `~/.ai-console-agent/task-journal.json`（0600），按 task ID 保存最近结果：

```json
{
  "task-id": {
    "action": "write_config",
    "status": "done",
    "result": {"backup":"...","content_sha256":"..."},
    "finished_at": 0
  }
}
```

收到已完成 task ID 的新 claim 时不重新执行，直接使用新 nonce 回报缓存结果。保留最近 1000 条或 30 天，原子写临时文件后 rename。

对 `upgrade_agent` 要在替换二进制和退出前先持久化完成结果；否则进程退出后任务会再次执行。

### 6.5 Agent 串行执行

`connectWS()` 不再为每条命令直接启动无界 goroutine。增加单 worker channel：

```go
cmdQueue := make(chan wsCmd, 64)

// reader 只负责解析和入队
cmdQueue <- cmd

// worker 顺序执行所有会改变机器状态的任务
for cmd := range cmdQueue {
    a.handleCmdWS(conn, &cmd)
}
```

heartbeat/ping 仍独立运行，WS 写继续使用现有 mutex。队列满时返回明确错误或暂时停止读取，不能无限创建 goroutine。

如果未来需要并发，最多只并发纯读 action；`write_config`、restore、credential、upgrade 必须使用同一 mutation mutex。

### 6.6 配置写入原子性

`handleWriteConfig()`：

1. 读取并写好 0600 backup；
2. 在同目录创建 0600 临时文件；
3. 写入、`Sync()`、关闭；
4. rename 覆盖正式配置；
5. fsync 父目录（支持的平台）；
6. 写入 task journal；
7. 回报结果。

### 6.7 验收

- 让旧 claim 超时后重新 claim，再提交旧 nonce 结果，必须 409 且任务保持新一轮 running。
- 模拟 8 分钟任务持续续租，不能被 reaper 回收。
- 断开结果上报网络后重新领取同一 task ID，Agent 返回 journal 结果且不产生第二份备份。
- 连续下发两个同工具 write_config，最终顺序与任务创建顺序一致。

## 7. BUG-06：Agent 发布构建入口缺失，Docker 分发陈旧二进制

### 7.1 新增 main 包

创建 `agent/cmd/ai-agent/main.go`：

```go
package main

import (
    "context"
    "flag"
    "fmt"
    "os"
    "os/signal"
    "syscall"

    agentlib "ai-console-agent/internal/agent"
)

var version = "dev"

func main() {
    server := flag.String("server", os.Getenv("SERVER"), "console base URL")
    token := flag.String("token", os.Getenv("TOKEN"), "one-time enroll token")
    enrollOnly := flag.Bool("enroll-only", false, "enroll and exit")
    showVersion := flag.Bool("version", false, "print version")
    flag.Parse()

    if *showVersion {
        fmt.Println(version)
        return
    }

    ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
    defer stop()

    a, err := agentlib.New(&agentlib.Config{Server: *server, Token: *token, Version: version})
    if err != nil { fatal(err) }
    if *enrollOnly {
        if err := a.Enroll(ctx); err != nil { fatal(err) }
        return
    }
    if err := a.Run(ctx); err != nil && err != context.Canceled { fatal(err) }
}
```

补 `fatal()`，且同时支持现有 installer 使用的 `--enroll-only --token --server` 与 upgrade 使用的 `--version`。

### 7.2 构建和发布

- `build-dist.sh` 的默认版本不再写死旧版本，要求参数或从 git tag/环境变量获得；
- 构建后逐个执行二进制 `--version` 并与 manifest.version 比对；
- CI 至少执行 `go test ./...`、`go vet ./...`、四平台交叉编译；
- 发布时把 manifest 和四个二进制作为同一不可分割 artifact。

Dockerfile 不应直接复制开发机上可能陈旧的 `console/agent-dist`。推荐 multi-stage：在 Go builder stage 从当前源码生成 agent-dist，再复制进 Node runtime image。若构建环境无法交叉编译，则 Docker 构建必须接收 CI 生成且带版本校验的 artifact，发现源码提交和 manifest 不匹配时失败。

### 7.3 验收

- `bash agent/build-dist.sh vX.Y.Z` 成功生成四个平台二进制；
- 四个二进制 `--version` 均为 `vX.Y.Z`；
- manifest size/hash 与文件一致；
- 新 Docker 镜像内 `/app/console/agent-dist/manifest.json` 版本与当前构建参数一致；
- `install.sh --enroll-only` 路径可实际运行。

## 8. BUG-07：Agent 安装/自升级未验证下载内容

### 8.1 最低安全基线

Agent 和 installer 默认只允许 HTTPS Console URL。仅显式开发参数允许 `http://localhost`/内网 HTTP，且日志打印高亮 warning。

升级步骤：

1. 获取 `/agent/manifest.json`；
2. 根据 `GOOS-GOARCH` 找到 version、size、sha256；
3. 使用带 connect/header/total timeout 的专用 `http.Client` 下载；
4. 禁止重定向到不同 host；
5. 用 `io.LimitReader` 限制最大体积；
6. 边下载边计算 SHA-256；
7. size/hash 均匹配后才 chmod/执行 `--version`；
8. `--version` 必须等于 manifest.version；
9. 原子替换并返回 `new_version: manifest.version`。

manifest 和 binary 来自同一 HTTPS 服务时，hash主要防损坏和发布错配；若需要抵御 Console 存储或发布链路被篡改，进一步为 manifest 添加 Ed25519 签名，并在 Agent 中内置发布公钥。私钥只能存在于 CI secret，不能进入仓库或 Console 容器。

### 8.2 installer

`install.sh` 下载 manifest 后用 `sha256sum` 或 `shasum -a 256` 校验。系统两者都不存在时失败退出，不能警告后继续。

自定义 `BINARY_URL` 必须同时提供预期 SHA256，例如 `BINARY_SHA256`，否则拒绝安装。

### 8.3 失败恢复

- 新二进制校验或 `--version` 失败：删除 `.new`，保留当前 binary；
- rename 新 binary 失败：立即恢复 `.bak`；
- 服务重启后 heartbeat 未在超时内报告新版本：Console 标记 upgrade task failed，并提示人工恢复 `.bak`；
- 手工运行、没有 service manager 时，不执行会 `os.Exit(0)` 的自升级，或要求显式 `--allow-unmanaged-restart`。

## 9. BUG-08：机器用 hostname+OS 去重导致互相顶号

### 9.1 身份模型

hostname 只能是展示信息，不能作为机器主键。Agent 首次启动时生成并持久化 `agent_instance_id` UUID，注册时发送。新增迁移 `014_agent_instance_identity.sql`：

```sql
ALTER TABLE servers ADD COLUMN agent_instance_id TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_servers_agent_instance_id
ON servers(agent_instance_id)
WHERE agent_instance_id IS NOT NULL;

ALTER TABLE agent_enroll_tokens ADD COLUMN target_server_id TEXT;
ALTER TABLE agent_enroll_tokens ADD COLUMN enroll_mode TEXT NOT NULL DEFAULT 'new';
```

### 9.2 注册规则

- 默认 `enroll_mode=new`：始终创建新 server 行，即使 hostname/OS 相同；
- Agent 带已知 `agent_instance_id` 且服务端已有对应行：只有明确 replace/recover token 才允许轮换 token；
- UI 的“重新安装/恢复此机器”生成带 `target_server_id`、`enroll_mode=replace` 的一次性 token；
- replace 时必须校验 token 的 target 与找到的 server 完全一致；
- 普通 enroll token 不能覆盖任何现有 server。

如果旧 Agent 没有 instance ID，升级后生成并通过 heartbeat 绑定当前 server；服务端只允许已认证 agent token 为自己的 server 首次写入该字段。

### 9.3 验收

- 两台 `hostname=devbox, os=linux` 的新 Agent 注册后得到两个不同 server ID，二者都保持在线；
- 普通 enroll token 无法轮换已有 server token；
- replace token 只能替换指定 server，不能更改其他机器；
- 克隆了 state.json 的机器触发 instance ID 冲突时拒绝上线并提示重新注册，不能静默抢占。

## 10. BUG-09：浏览器 WS 自动重连会清空所有订阅

### 10.1 拆分“断开连接”和“销毁客户端状态”

当前 `initWS()` 调 `closeWS()`，而 `closeWS()` 会清空 `subscribers`。改为：

```ts
function disconnectSocket() {
  // 只清 timer、handler、socket，不清 subscribers
}

export function closeWS() {
  currentToken = null
  disconnectSocket()
  subscribers.clear()
  pendingSubscriptions.clear()
}
```

`initWS()` 和自动重连只调用 `disconnectSocket()`；只有 logout/AuthProvider 卸载才调用完整 `closeWS()`。

### 10.2 防旧 socket 事件破坏新连接

创建 socket 后保存局部变量和 generation：

```ts
const generation = ++connectionGeneration
const socket = new WebSocket(url)
ws = socket

socket.onclose = () => {
  if (ws !== socket || generation !== connectionGeneration) return
  ws = null
  scheduleReconnect()
}
```

这与 Agent 端 current-socket close guard 是同类竞态，属于 load-bearing 逻辑。

### 10.3 ticket single-flight

single-flight 只应共享“尚未完成的领票请求”。Promise settle 后立即设为 null，不能额外缓存 1 秒，因为 ticket 是一次性资源；第一次连接消费后，第二次 init 会复用已消费 ticket并收到 4001。

### 10.4 验收

- 打开服务器列表后强制关闭 WS，重连后 `subscribers.size` 不变，服务器事件立即触发 query invalidation；
- 连续快速执行两次 init，第二个连接使用新 ticket；
- 旧 socket 的延迟 close 事件不能清掉新 socket；
- logout 后 subscriber map 必须清空且不再重连。

## 11. BUG-10：迁移版本领养错误跳过 006–008

### 11.1 不再无条件标记 drop/data migration

为 `detectAppliedMigrationVersions()` 增加 `hasTable()`，按真实 footprint 判断：

```ts
const hasTable = (table: string) => Boolean(
  db.prepare("SELECT 1 FROM sqlite_master WHERE type='table' AND name=?").get(table)
)

const migration006Applied =
  !hasCol("users", "totp_secret") &&
  !hasCol("users", "totp_enabled") &&
  !hasCol("users", "recovery_codes_hash") &&
  !hasTable("usage_snapshots") &&
  !hasTable("bindings") &&
  !hasTable("profiles")

if (migration006Applied) out.push("006")
else return out

if (!hasTable("test_runs")) out.push("007")
else return out
```

008 是可重复执行的脱敏 UPDATE，不需要猜测是否已执行。若迁移记录缺失，让迁移 runner 重新执行 008；其 WHERE 条件应扩展到 BUG-01 的历史敏感字段，且保证幂等。

### 11.2 正确处理只应用了 009 的库

当前只识别“010 已完成”的 footprint。增加：

```ts
if (hasCol("provider_keys", "default_model_id")) {
  out.push("009", "010")
} else if (hasCol("providers", "default_model_id")) {
  out.push("009")
}
```

否则只应用过 009 的库会重新执行 009 并 duplicate column。

### 11.3 不只处理 applied.size===0

数据库可能已有 `001_baseline` 记录、但后续记录丢失。将 footprint reconciliation 抽成独立步骤，对每个缺失 migration：

- 有可靠正向 footprint 时补记 applied；
- data-only 且幂等时直接重跑；
- 无法证明时停止启动并输出修复命令，不能猜测版本；
- 所有补记在事务内完成并写结构化日志。

### 11.4 迁移矩阵测试

新增 `console/db/migration-smoke.cjs`，在临时目录创建以下库形状：

| 起始形状 | schema_migrations | 预期 |
|---|---|---|
| 空库 | 空 | 001→最新全部执行 |
| 005 | 空 | 006/007/008 必须执行 |
| 006 | 仅 001 | 007/008→最新执行 |
| 009 | 001–008 | 只执行 010→最新 |
| 010 | 空 | 正确领养 001–010，只执行 011+ |
| 011 | 空 | 全部领养，零 DDL 重放 |
| 011 | 仅 001 | reconciliation 后正常启动 |

每个 case 启动两次，第二次必须零迁移、无 duplicate column。测试完成后删除精确临时目录，不操作真实 `console/data`。

## 12. 跨问题发布与兼容策略

### 12.1 Console/Agent 协议版本

heartbeat 增加 `protocol_version`。Console 在 servers 表保存它，并按版本决定是否发送 nonce-required 任务。推荐：

```text
protocol 1：旧协议，无结果 nonce（仅过渡期）
protocol 2：nonce + lease + task journal
```

新增迁移 `015_agent_protocol_version.sql` 并同步 `schema.sql`：

```sql
ALTER TABLE servers ADD COLUMN agent_protocol_version INTEGER NOT NULL DEFAULT 1;
```

heartbeat 缺少该字段的旧 Agent 按 protocol 1 处理，不能错误地推断为最新版。

发布顺序：

1. Console 先兼容 protocol 1/2，但对 protocol 1 暂停自动 requeue 非幂等任务；
2. 发布可构建的新 Agent protocol 2；
3. 控制台观察全部活跃 Agent 完成升级；
4. 开启 `MIN_AGENT_PROTOCOL=2` 强校验；
5. 删除无 nonce 兼容分支。

### 12.2 数据库备份

执行 012–015 前：

- 停止 Console 写入；
- 使用 SQLite backup API 或 `.backup` 生成一致性备份，不能只复制 WAL 主文件；
- 记录当前 `MASTER_KEY` 的指纹而不是原文；
- 在备份副本上先跑 migration-smoke 和历史敏感数据迁移；
- 验证后再升级生产库。

### 12.3 回滚发布

新增列均向后兼容，但新 Console 生成的引用型任务旧 Console 无法理解。应用回滚时必须保证：

- 先停止创建新格式任务；
- 等待 pending/running 任务清空；
- 再回滚 Console；
- 不回滚已执行的数据脱敏，敏感明文不应恢复。
